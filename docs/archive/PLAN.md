# План улучшений ARCHON

Обзор системы, выявленные проблемы и рекомендации по рефакторингу.

---

## 1. Общая картина

- **Backend:** FastAPI, роуты подключаются из `app.api` через `routes.py`, общий слой — `app.services.knowledge` и репозитории.
- **Frontend:** React + React Router, Zustand, React Query, много страниц внутри вселенной (персонажи, главы, чат и т.д.).
- **Связность:** Единый API-клиент, выбор вселенной через store и URL, чат/RAG/таймлайн опираются на контекст вселенной.

## 4. Backend: централизация LLM-настроек

**Проблема:** В нескольких местах повторяется один и тот же код: чтение `system_settings` и подстановка `openrouter_api_key`, `routerai_*`, `get_default_model` в `llm_service` (chat.py, characters.py, locations.py, ai_critic.py, universe_view.py).

**Решение:** Один хелпер, например:
- `app.services.llm_config.load_llm_settings_from_system(system_settings, provider?)`
Вызывать в начале каждого эндпоинта, использующего LLM. Так не будет расхождений при добавлении нового провайдера или поля.

## 5. API книг и вселенных

- **universes.py** — основной API вселенных.
- **books.py** — дублирует те же операции под префиксом `/books` для обратной совместимости.
- Фронт использует только `universesApi` (префикс `/api/universes`).

**Рекомендация:** Если нет внешних клиентов на `/api/books`, пометить books как deprecated и со временем удалить; иначе оставить только для тестов и явно задокументировать.

## 6. Связность сущностей (оставить как есть, при желании усилить)

| Связка | Состояние | Идея |
|--------|-----------|------|
| **Outline ↔ Chapters** | OutlineItem имеет `chapter_id`. Логика связная. | На странице плана показывать привязку к главе и переход; в главе — «пункт плана: …». |
| **Coverage (chapter_mention)** | Используется для аналитики «кто/что в каких главах». API и репозиторий согласованы. | На странице главы показывать список упоминаний; при необходимости — на странице персонажа/локации «в каких главах фигурирует». |
| **Чат, RAG, знания персонажей** | Чат использует RAG и опционально знания персонажа, время диалога привязано к вселенной. | Подключить RAGContextDisplay в интерфейсе чата или удалить компонент. |

## 7. Навигация и названия

- **«Писать» vs «Написать книгу»:** разная логика (редактор по главам vs пошаговый мастер). Имеет смысл оставить оба, но в меню явно подписать: «Писать» (по главам) и «Мастер книги».
- **Два пункта «Настройки»:** в сайдбаре внутри вселенной — «Настройки системы» (`/settings`), в подменю «Служебное» — «Настройки вселенной» (UniverseConfigPage). Различать подписи: «Настройки системы» и «Настройки вселенной».

---

## 8. Идеи на будущее (опционально)

- **Единая точка входа для AI по сущности:** эндпоинт вида `POST /universes/{id}/entities/{type}/{entity_id}/ai` с телом `{ "action": "analyze" | "suggest_improvements" | "generate_related" }`, внутри вызывающий ai_critic или ai_generator.
- **Использование RAG при генерации глав:** при вызове ai_writer подмешивать релевантные фрагменты из RAG в системный промпт (если ещё не используется).
