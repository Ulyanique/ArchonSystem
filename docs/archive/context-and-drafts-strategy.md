# Контекст и черновики: только релевантная информация в бюджет

## Проблема

В базе много неструктурированных черновиков с повторами. Раньше в контекст чата подставлялись **все** черновики подряд (до лимита символов), из‑за чего:
- в контекст попадала нерелевантная информация;
- бюджет токенов тратился впустую;
- большая часть данных вообще не помещалась в контекст.

## Решение на бэкенде (сделано)

### 1. Только релевантный контекст по умолчанию

- **Раньше:** в системный контекст всегда добавлялся блок «ЧЕРНОВИКИ» — подряд все черновики до `draft_context_budget` символов.
- **Теперь:** по умолчанию этот блок **не** добавляется. В контекст попадают только:
  - **Поиск по запросу** (`SearchService`) — по словам запроса находятся заметки/черновики, и в промпт подставляется полный текст найденных (блок «ТЕКСТ НАЙДЕННЫХ ЧЕРНОВИКОВ/ЗАМЕТОК»).
  - **RAG** (ChromaDB + эмбеддинги) — семантический поиск по запросу; в контекст идут только релевантные фрагменты глав и заметок.

Так мы укладываемся в бюджет и убираем повторы: модель видит только то, что относится к запросу.

### 2. Опция «все черновики в контекст»

Если нужен старый режим (все черновики подряд), в настройках системы можно включить:

- **`include_all_drafts_in_context`** (в БД: `system_settings.include_all_drafts_in_context`).

При `True` блок «ЧЕРНОВИКИ» снова добавляется в контекст, как раньше. По умолчанию `False`.

### 3. Черновики в RAG чанками

- Заметки и черновики в RAG теперь хранятся **чанками** (как главы): разбиение по ~1500 символов с перекрытием.
- В контекст попадают только **релевантные чанки** по запросу, а не целый длинный черновик.
- Для уже существующих заметок индексация обновится при следующем сохранении; при желании можно запустить переиндексацию всех заметок вселенной.

## Поиск и структурирование на стороне клиента

Чтобы ещё лучше уложиться в бюджет и передавать нейросети только нужное:

### 1. Предпросмотр контекста перед отправкой

- Перед отправкой сообщения в чат вызывать **поиск по текущему тексту запроса** (например `GET /api/universes/{id}/search?q=...` или аналог).
- Показать пользователю: «По запросу будут использованы: заметка X, черновик Y, глава Z».
- Опционально: чекбоксы «включить/исключить» источники, чтобы пользователь сам резал контекст (если API это поддерживает).

### 2. Структурирование черновиков

- **Теги/типы:** уже есть `note_type` (idea, research, draft, avoid, other). Имеет смысл использовать их последовательно и при необходимости ввести подтипы или теги.
- **Заголовки и блоки:** если редактор черновиков поддерживает заголовки (например Markdown `##`), бэкенд может при индексации резать по ним и класть в RAG отдельные чанки — релевантным окажется только нужный блок.
- **Краткое описание:** поле «краткое содержание» или «тезисы» у черновика даёт возможность подставлять в контекст только его вместо всего текста, когда полный текст не нужен.

### 3. Явный выбор источников в UI

- В форме чата: «Добавить в контекст: [поиск по запросу]» с списком найденных заметок/глав и возможностью снять галочки.
- Так пользователь сам решает, что именно попадёт в контекст, и укладывается в бюджет.

## Итог

| Место | Что сделано / что делать |
|--------|---------------------------|
| **Бэкенд** | По умолчанию в контекст не подставляются все черновики; только релевантные по поиску и RAG. Черновики в RAG хранятся чанками. Опция `include_all_drafts_in_context` возвращает старый режим при необходимости. |
| **Клиент** | Имеет смысл: предпросмотр «что попадёт в контекст» по запросу, структурирование черновиков (теги, заголовки), опционально явный выбор источников контекста перед отправкой. |

В результате в контекст попадает в основном релевантная информация, бюджет используется эффективнее, а дубли и лишние черновики не занимают место.
