# RAG и управление контекстом чата

RAG (Retrieval Augmented Generation) — система векторного поиска, которая enriches контек чата релевантными фрагментми из глав и заметок вселенной.

## Архитектура RAG

```
┌─────────────────────┐
│   Запрос пользователя │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   ChromaDB поиск    │  sentence-transformers эмбеддинги
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Knowledge Filter   │  Фильтрация по уровню доступа персонажа
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Temporal Filter    │  Исключение «будущих» событий
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Budget Capping     │  Ограничение по символам
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Системный промпт   │  Финальный контекст для LLM
└─────────────────────┘
```

## Векторный поиск

### ChromaDB

- **Модель эмбеддингов**: `paraphrase-multilingual-MiniLM-L12-v2` (по умолчанию)
- **Настраивается**: `RAG_EMBEDDING_MODEL` в `.env`
- **Мультиязычная**: хорошо работает с русским текстом

### Чанкинг

Текст разбивается на фрагменты для индексации:

| Параметр | Значение |
|----------|----------|
| Размер чанка | 1500 символов |
| Overlap | 300 символов |
| Источники | Главы (`content`), заметки (`content`) |

### Изоляция

Каждая вселенная имеет **отдельное** векторное пространство:
- Поиск фильтруется по `universe_id`
- Переиндексация — только для одной вселенной
- Удаление вселенной = удаление её индекса

### Переиндексация

```
POST /api/universes/{id}/rag/reindex
```

Переиндексирует все главы и заметки вселенной. Выполняется автоматически при:
- Создании/обновлении главы
- Создании/обновлении заметки

Или вручную через эндпоинт.

## Уровни знаний персонажа

Персонаж не знает всё о вселенной. Система моделирует **ограниченное знание** через уровни доступа.

### Уровни (от низшего к высшему)

| Уровень | Значение | Описание |
|---------|----------|----------|
| `none` | 0 | Персонаж ничего не знает |
| `rumors` | 1 | Слышал смутные слухи |
| `superficial` | 2 | Поверхностное понимание |
| `good` | 3 | Хорошее, детальное знание |
| `complete` | 4 | Полное, исчерпывающее знание |

### Источники знаний

| Источник | Описание |
|----------|----------|
| `participated` | Персонаж лично участвовал в событии |
| `witnessed` | Персонаж был свидетелем |
| `heard` | Слышал от других персонажей |
| `read` | Прочитал в документах/книгах |
| `learned_from` | Изучил из другого источника |

### Применение в чате

Когда пользователь пишетет персонажу:

1. **RAG находит** релевантные события/главы
2. **Knowledge Filter** проверяет: «Знает ли персонаж это?»
   - Есть ли запись в `character_knowledge`?
   - Достаточно ли высокий `knowledge_level`?
3. **Temporal Filter** проверяет: «Не из будущего ли это?»
   - Сравнивает дату события с датой диалога
4. **Результат** подставляется в системный промпт

## Структура системного промпта

Промпт формируется в **фиксированном порядке** для Prefix Caching в Ollama:

```
┌─────────────────────────────────────────┐
│  [1] Статичный мир                      │  ← не меняется между запросами
│     - Описание вселенной                │
│     - Стиль, жанр, правила мира         │
│     - Настройки промпта                 │
├─────────────────────────────────────────┤
│  [2] Персонаж                           │  ← меняется редко
│     - Имя, характер, внешность          │
│     - Паттерн речи                      │
│     - Текущий момент в времени          │
├─────────────────────────────────────────┤
│  [3] Динамические данные                │  ← меняется каждый запрос
│     - RAG-найденные фрагменты           │
│     - Упоминания из текста [[type:id]]  │
│     - История диалога                   │
└─────────────────────────────────────────┘
```

### Prefix Caching

Ollama кэширует префикс запроса. Стабильный порядок означает:
- Блоки [1] и [2] **кэшируются** между запросами
- Только блок [3] пересчитывается
- **Экономия** 30-50% времени генерации

## Бюджетирование символов

Каждый блок контекста имеет лимит символов для предотвращения превышения контекстного окна LLM.

### Настраиваемые лимиты

Хранятся в `system_settings.chat_context_budget`:

| Блок | Лимит (по умолчанию) |
|------|---------------------|
| Описание мира | 2000 символов |
| Описание персонажа | 3000 символов |
| RAG-фрагменты | 4000 символов |
| История диалога | 6000 символов |
| Динамические упоминания | 2000 символов |

### Алгоритм обрезки

Если блок превышаетет лимит:
1. Текст обрезается до лимита
2. Добавляется пометка `... [обрезано]`
3. Приоритет: более релевантные фрагменты сохраняются

## Обнаружение упоминаний

Система парсит текст на маркеры `[[entity_type:entity_id]]`:

```
В тексте главы есть упоминание: [[character:uuid]] вошёл в [[location:uuid]].
```

При чате:
1. Парсинг маркеров из RAG-фрагментов
2. Загрузка полных данных об сущностях
3. Подстановка в контекст промпта

Утилита `resolve_markers.py` разрешает маркеры в реальные имена сущностей.

```env

```

```
PUT /api/system/settings

  "chat_context_budget": {
    "world_description": 2000,
    "character_description": 3000,
    "rag_fragments": 4000,
    "dialog_history": 6000,
    "dynamic_mentions": 2000
  }
}
```
